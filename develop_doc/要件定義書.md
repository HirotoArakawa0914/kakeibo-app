## 1. プロジェクト概要

### 1-1. 目的

本プロジェクトは「家計簿アプリ」の開発を通じて、以下4点の学習・実績化を主目的とする。

- CI/CD パイプラインの設計・構築経験の習得
- Ruby on Rails による Web アプリ開発スキルの向上
- OCR を利用したアプリ開発経験の獲得
- 転職・就職活動で提示できるポートフォリオ成果物の作成

> ※「動けばいい」ではなく、学習効果・保守性・ポートフォリオ価値の高い設計を優先する。

### 1-2. 開発方針

| 観点 | 方針 |
|---|---|
| 開発手法 | アジャイル開発（2週間スプリント） |
| 実装順 | MVP → カテゴリ → 集計/グラフ → OCR → AI → 認証 の順で拡張 |
| 品質基準 | CI（GitHub Actions）でテスト通過をマージ条件とする |
| デプロイ | main マージで Render.com へ自動デプロイ |
| スコープ管理 | MVP 優先・1スプリント（2週間）で実装可能な単位に分割 |

## 3. 機能要件

### 3-1. フェーズ構成

| フェーズ | スプリント | 主要機能 | 優先度 |
|---|---|---|---|
| Phase 1 | S1〜S2 | 収支CRUD・CI/CD構築 | 最高 |
| Phase 2 | S3〜S4 | カテゴリ管理・集計・グラフ | 高 |
| Phase 3 | S5〜S6 | OCR読取・補正UI | 高 |
| Phase 4 | S7〜S8 | AIアドバイス | 中 |
| Phase 5 | S9〜 | 認証・マルチユーザー化 | 低 |

### 3-2. Phase 1 ― 収支CRUD（MVP）

- 日付・金額・種別（収入/支出）・カテゴリ・メモを登録
- 一覧表示・詳細表示・編集・削除
- 入力バリデーション（必須項目・数値チェック等）
- GitHub Actions による CI（自動テスト）
- Render.com への自動デプロイ

### 3-3. Phase 2 ― カテゴリ・集計・グラフ

**カテゴリ管理**

- カテゴリマスタの CRUD（食費・交通費・娯楽 等）
- カテゴリカラー設定（グラフ表示用）

**集計・グラフ**

- 月別支出合計（棒グラフ）
- カテゴリ別内訳（円グラフ）
- 収支バランス（棒グラフ：収入 vs 支出）
- 日別推移（折れ線グラフ）

### 3-4. Phase 3 ― OCR読取・補正UI

- レシート画像のアップロード
- OCR によるテキスト抽出（店名・日付・品目・金額）
- 抽出結果の手動修正 UI（誤認識前提の設計）
- 修正後に収支データとして登録
- OCR処理は非同期で実行（UX確保）

> ※精度より「修正しやすさ」を重視した UI 設計とする。

### 3-5. Phase 4 ― AIアドバイス

- 月次支出データをもとに支出傾向を文章生成
- OpenAI API（または Claude API）と連携
- ダッシュボードにアドバイスパネルとして表示

### 3-6. Phase 5 ― 認証・マルチユーザー化

- Devise による認証機能追加
- user_id を transactions に紐付け（DB設計は最初から考慮）
- ログイン・ログアウト・サインアップ

## 4. 非機能要件

| 項目 | 要件 |
|---|---|
| レスポンシブ対応 | スマホでレシート撮影・登録ができるよう必須対応 |
| テスト | Rails 標準（Minitest）でモデルテスト・システムテストを実装 |
| セキュリティ | CSRF 対策・ストロングパラメータ（Rails 標準機能で対応） |
| パフォーマンス | N+1 クエリを発生させない（includes を適切に使用） |
| 認証（初期） | フェーズ5まで不要。ただし DB 設計は user_id を考慮した構成とする |
| 可用性 | Render.com 無料プランのコールドスタートは許容 |
| ログ | Rails デフォルトログで対応（本番は Render.com のログ画面で確認） |

## 5. データ設計

### 5-1. テーブル一覧

| テーブル名 | 概要 | 追加タイミング |
|---|---|---|
| categories | カテゴリマスタ（食費・交通費 等） | Phase 2 |
| transactions | 収支データの本体 | Phase 1（MVP） |
| receipts | OCR処理結果の格納 | Phase 3 |
| users | 認証ユーザー情報 | Phase 5 |

### 5-2. ER図

```
users (Phase 5)
  id, email, password_digest, created_at, updated_at
  |
  | 1:N
  |
categories (Phase 2)
  id, name, color, icon, user_id(null許容), created_at, updated_at
  |
  | 1:N
  |
transactions (Phase 1 ★MVP)
  id, user_id(null許容), category_id,
  transaction_type(income/expense),
  amount, date, memo,
  receipt_image, ocr_raw_text, ocr_status,
  created_at, updated_at
  |
  | 1:1
  |
receipts (Phase 3)
  id, transaction_id, image_path,
  raw_text, parsed_data(JSON),
  status(pending/done/failed),
  created_at, updated_at
```

### 5-3. transactions テーブル定義

| カラム名 | 型 | 説明 |
|---|---|---|
| id | integer | 主キー（自動採番） |
| user_id | integer | 外部キー（Phase5 まで null 許容） |
| category_id | integer | 外部キー → categories |
| transaction_type | string | `"income"` / `"expense"` |
| amount | integer | 金額（円） |
| date | date | 取引日付 |
| memo | string | メモ（任意） |
| receipt_image | string | レシート画像パス（Phase3 以降） |
| ocr_raw_text | text | OCR 抽出テキスト（Phase3 以降） |
| ocr_status | string | `pending` / `done` / `failed`（Phase3 以降） |
| created_at / updated_at | datetime | Rails 標準タイムスタンプ |

### 5-4. categories テーブル定義

| カラム名 | 型 | 説明 |
|---|---|---|
| id | integer | 主キー |
| name | string | カテゴリ名（食費・交通費 等） |
| color | string | HEX カラーコード（グラフ用） |
| icon | string | アイコン識別子（任意） |
| user_id | integer | 外部キー（Phase5 まで null 許容） |
| created_at / updated_at | datetime | Rails 標準タイムスタンプ |

### 5-5. receipts テーブル定義

| カラム名 | 型 | 説明 |
|---|---|---|
| id | integer | 主キー |
| transaction_id | integer | 外部キー → transactions |
| image_path | string | アップロード画像のパス |
| raw_text | text | OCR 生テキスト |
| parsed_data | json | 構造化抽出結果（店名・日付・金額 等） |
| status | string | `pending` / `done` / `failed` |
| created_at / updated_at | datetime | Rails 標準タイムスタンプ |

## 9. CI/CD 構成

### 9-1. ブランチ戦略

| ブランチ | 用途 | 直接 push |
|---|---|---|
| `main` | 本番環境（Render.com） | 禁止（PR のみ） |
| `develop` | 開発統合ブランチ | 基本禁止 |
| `feature/*` | 機能ごとの開発ブランチ | OK |

### 9-2. GitHub Actions ワークフロー概要

```
push / PR (main・develop)
  │
  ▼
[test ジョブ]
  SQLite で DB 作成
  → bundle exec rails db:migrate
  → bundle exec rails test
  │
  └─ 成功 & main への push のみ
       │
       ▼
     [deploy ジョブ]
       Render API へ POST
       → 自動デプロイ実行
```

### 9-3. 環境変数管理

| 場所 | キー | 内容 |
|---|---|---|
| Render.com | `DATABASE_URL` | PostgreSQL 接続 URL（`postgres://` 始まり） |
| Render.com | `RAILS_ENV` | `production` |
| Render.com | `RAILS_MASTER_KEY` | config/master.key の内容 |
| Render.com | `RAILS_SERVE_STATIC_FILES` | `true` |
| GitHub Secrets | `RENDER_API_KEY` | Render の API キー |
| GitHub Secrets | `RENDER_SERVICE_ID` | Render のサービス ID |
